AI-Native SDLC

A proposal to replace ticket-centric agile with a spec-driven, agent-observable development lifecycle.
Draft for Review April 2026 Status: Experimental

Why change?

Jira-era agile was designed for humans writing tickets, humans doing work, and humans reading dashboards. AI agents break every assumption in that model:

What broke

  • Tickets are the root artifact. But agents need specs — structured intent they can read, not prose tickets with context in comments and tribal knowledge.
  • Humans are the only assignees. Agents have no identity, budget, or audit trail in Jira. Their work is invisible.
  • Work is a status on a card. Agent work is a run — prompt, tool calls, cost, evals, diffs. None of this is capturable in a ticket status.
  • Reporting assumes human throughput. Story points and velocity don't measure agent cost-per-feature, regression rate by author type, or defect density per spec.

What we need instead

  • Spec as root. Versioned, structured, machine-readable. The durable artifact that both humans and agents treat as source of truth.
  • Agents as first-class assignees. With capabilities, constraints, budgets, and observable execution traces.
  • Runs as a primitive. Every agent execution captured: what it did, what it cost, whether it passed evals, what artifacts it produced.
  • Role-specific views. Execs see initiative rollups. PMs see spec status. Devs see task queues. All are projections over the same graph.

Five principles

PrincipleImplication
Spec is the root, not the ticketTickets are ephemeral projections. Specs are versioned markdown in the repo, linked from the tracker.
Agents are assignees, not toolsThey have identities, budgets, audit trails. Assign a task to an agent the same way you assign it to a person.
Runs are first-classEvery agent execution is a node in the work graph: cost, tool calls, evals, artifacts. This is how you get observability.
Views are projectionsBoards, timelines, dashboards are queries — not schema. Stop forcing everyone into the same Kanban board.
Bugs are spec violationsNot a parallel system. A bug is a signal that reality disagrees with a prior spec. Same graph, same pipeline.

The work graph

Everything is a typed node with typed edges. The graph is event-sourced — append-only log with materialized views.

Initiative — business outcome, owner=human, horizon=quarters
  └─ Spec — versioned markdown, the "why + what"
      ├─ Plan — decomposition into tasks, can be agent-drafted
      │    └─ Task — unit of execution, assignee = human | agent
      │         └─ Run — one agent execution: cost, tools, evals, artifacts
      └─ ADR — architecture decisions linked to the spec
The key insight: Jira has no equivalent of a Run. Agent work is invisible in ticket-based systems. Runs make it observable — cost per feature, regression per agent, eval trends over time.

Edge types

EdgeConnectsEnables
decomposesPlan → SpecTrace any task back to the intent it serves

regression-ofBug Spec → RunWhich agent/model/run introduced this bug?

What the graph unlocks (vs. Jira)

Jira can answer

  • How many tickets closed this sprint?
  • What's our velocity trend?
  • Who has the most open tickets?

The work graph can answer

  • What's the cost per feature, from spec to deploy?
  • Which specs have the most defects?
  • Do agent-authored features regress more than human-authored?
  • Which agent/model produces the most durable code?
  • How long from user-reported signal to merged fix?

The spec layer

The entire framework says "spec is the root" — so we can't defer how specs work. Specs are schema-enforced markdown in the repo, validated in CI, indexed for agents.

Why not an external tool?

Specs must version with the code they describe. A spec updated in the same PR as the code change is reviewable, atomic, and doesn't require sync between systems. External tools (Notion, Kiro, Spec Kit) put specs outside the repo — creating sync drift, a second source of truth, and another integration agents must bridge.

Spec anatomy

YAML frontmatter (structured, parseable)

id: SPEC-001 ← immutable, referenced everywhere
title: "User auth flow"
status: draft → active → superseded
version: 1 ← bumps on material changes
initiative: INI-003 ← links to Linear
owner: franklin ← human accountable
tags: [auth, security]
linear_project: PRJ-XYZ

Body sections (required)

SectionPurpose
ProblemWhat's wrong, why it matters
Success criteriaMeasurable outcomes
ScopeIn / explicitly out
DesignArchitecture, linked ADRs
Acceptance criteriaTestable conditions (Given/When/Then)
Risks & constraintsWhat could go wrong
MigrationFor refactors: current → target → strategy → rollback

Three artifact types, one system

ArtifactID formatKey fieldsExample
SpecSPEC-NNNstatus, version, initiative, owner, acceptance criteriaspecs/SPEC-001-user-auth.md

How it works in practice

1. Author — copy template, fill frontmatter + sections, open PR
2. Validate — CI checks: frontmatter schema, required sections, reference resolution, status transitions
3. Review — team reviews the spec like code — intent, scope, acceptance criteria
4. Index — CI auto-generates spec-index.json — agents read this instead of scanning
5. Link — create Linear project, set linear_project field, bidirectional link established
6. Evolve — spec changes go through PRs. Version bumps on material changes. Old versions are superseded, never deleted.
Agent workflow: agent reads spec-index.json → finds the active spec for its task → reads the full spec → uses acceptance criteria as a checklist → cites criteria in PR comments. During triage, the NOC agent searches the index to find which spec a bug violates.

Agent architecture: Claude Code + Jules

Two agents with complementary execution models. Claude Code is the brain — it plans, orchestrates, and reviews. Jules is the workforce — it executes well-scoped tasks in parallel in the cloud.

DimensionClaude CodeJules
Runs whereLocal, in your terminalCloud VM (Ubuntu), clones your repo
ModeSynchronous — you steer in real-timeAsynchronous — fire and forget
ContextFull: local env, MCP (Linear, Slack), env vars, running servicesRepo clone only. No MCP, no local env
OutputEdits your working treeCreates branch + PR on GitHub
ConcurrencyOne sessionUp to 15 (Pro) / 60 (Ultra) concurrent

The spec is the contract between them

Jules has no MCP, no Linear, no local context. Everything it needs must be in the task prompt or in the repo. Claude Code assembles the prompt by extracting the relevant spec section, acceptance criteria, and constraints — then fires it to Jules via API. The spec layer is what makes this handoff seamless.

How Claude Code calls Jules

Jules exposes a REST API. Claude Code calls it directly from your terminal to create tasks, check status, and review output.

# Create a Jules task with auto-PR
curl -X POST julius.googleapis.com/v1alpha/sessions \
  -H "X-Goog-Api-Key: $JULES_API_KEY" \
  -d '{ "prompt": "... spec section + acceptance criteria ...",
       "sourceContext": { "source": "sources/ID" },
       "automationMode": "AUTO_CREATE_PR",
       "requirePlanApproval": false }'

# Or via CLI
jules remote new --repo owner/repo --session "task prompt"

Jules-eligible checklist

Claude Code assesses each task before routing:

Send to Jules if

  • Clear acceptance criteria in the spec
  • Self-contained — no env/infra/MCP deps
  • Has a test to verify, or task IS writing tests
  • Mechanical / parallelizable
  • Doesn't require architecture decisions

Keep in Claude Code if

  • Needs local env, credentials, or running services
  • Requires MCP (Linear, Slack, monitoring)
  • Requires interactive debugging or exploration
  • Involves architecture or judgment calls
  • Multi-step refactor with cascading decisions

The NOC agent: always-on intake

A dedicated agent monitors all signal sources continuously. Think of it as a Network Operations Center staffed by an agent with human escalation.

Signal sourceHow it arrivesAgent action
User reports (support, in-app, email)Webhook / queue from support toolParse, ask clarifying questions, draft Bug Spec
Error monitoring (Sentry, Datadog)Alert webhookCorrelate with recent deploys, check frequency/trend, auto-classify severity
Eval regressionsCI pipeline eventLink to the run + PR that introduced it, flag as regression
Slack / internal reportsChannel listener (MCP)Extract signal from conversation noise, normalize into Bug Spec
Uptime / health checksMonitoring alertsImmediate sev1 escalation if threshold hit, begin correlation

User reports: closing the loop

The highest-value integration. When a user reports a bug:

  1. NOC agent receives the signal — from support tool, in-app feedback, or email.
  2. Agent asks for context — replies to the reporter with specific questions (device? steps? when did it start?) rather than generic intake forms.
  3. Agent normalizes — drafts a Bug Spec with repro steps, linked spec, confidence scores. Cites the user's exact words.
  4. Agent attempts reproduction — runs the repro in a sandbox, attaches results as evidence.
  5. Human confirms in <60 seconds — reviews the pre-built spec, approves/edits.
  6. Reporter gets notified — "We've confirmed this and it's prioritized" or "This is working as designed because [spec link]." Fast, specific, respectful.
The NOC agent transforms triage meetings. By the time humans sit down, every bug has a normalized spec, reproduction attempt, severity proposal, and linked spec. The meeting is 100% prioritization, 0% data entry. Target: <60 seconds of human time per bug before the prioritization step.

Escalation guardrails

The NOC agent must hand off to a human immediately for:

  • Security vulnerabilities
  • Data loss or corruption
  • Payment / billing issues
  • Anything touching sev1 components
  • Affecting >N% of users (define N per product)
  • Agent confidence below threshold on severity

Codification: how agents learn the process

The SDLC only works if agents understand it. Process knowledge is codified in the repo so it's portable, versionable, and agent-agnostic.

Three-tier architecture

.ai/
├── sdlc.md    ← shared process (all agents read this)
├── CLAUDE.md ← local agent config (MCP, Linear, Jules orchestration)
└── AGENTS.md ← cloud agent config (VM constraints, PR conventions)
FileWho reads itContains
.ai/sdlc.md All agents Spec system, work tracker conventions, task lifecycle, boundaries, escalation rules. The portable process definition.
.ai/CLAUDE.md Claude Code (or any local agent) MCP integrations, Jules API orchestration, labeling workflow, phase responsibilities, daily summary format.
.ai/AGENTS.md Jules (or any cloud agent) VM environment, how to find specs from repo, PR conventions (branch naming, commit format, description template).
Agent portability: if you switch from Claude Code to Gemini CLI, adapt .ai/CLAUDE.md to the new agent's config format. The process in sdlc.md and Jules config in AGENTS.md stay unchanged. The process knowledge is portable — only the wiring changes.

Pilot plan

We'll test this on an upcoming major refactoring effort before proposing broader adoption.

Phase 1 — Setup

  • Copy .ai/ directory into the target repo (sdlc.md, CLAUDE.md, AGENTS.md)
  • Create specs/ directory with templates and CI validation
  • Create Linear workspace: initiative + project, labels (jules, claude-code, human)
  • Configure Claude Code with Linear MCP access
  • Generate Jules API key, store as JULES_API_KEY
  • Grant Jules access to the repo on GitHub
  • Customize AGENTS.md with project structure and setup commands

Phase 2 — Write the refactoring spec

  • Author specs/SPEC-001-[name].md with full frontmatter + body sections
  • Include migration section: current state → target state → strategy → rollback
  • Open a PR for the spec — review with the team
  • On approval: set status to active, create Linear project

Phase 3 — Plan and execute

  • Claude Code decomposes spec into Linear issues with routing labels
  • jules-labeled tasks dispatched via API — run in parallel in the cloud
  • claude-code-labeled tasks implemented locally with full context
  • Claude Code reviews all PRs (its own + Jules's) against the spec
  • Human does final review and merge
  • Run telemetry logged on each Linear issue

Phase 4 — Evaluate

  • Did specs-as-root improve clarity vs. tickets?
  • Did the Claude + Jules split reduce cycle time?
  • Was the .ai/ codification sufficient for agents to operate independently?
  • How did triage time compare to the old process?
  • What tooling gaps blocked us? What would we change?
